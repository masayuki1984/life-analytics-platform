# /update-docs コマンド

フィーチャー実装後や設計変更後に `docs/` 永続ドキュメントと CLAUDE.md を最新化するコマンド。

**使い方**: `/update-docs`  
**目的**: コードの実態とドキュメントのズレを解消し、次の `/add-feature` が正しい文脈で動けるようにする

---

## 重要ルール

- **更新対象ドキュメントの確認をユーザーに取ってから変更する**（サイレント上書き禁止）
- 1 ファイルずつ更新し、ユーザーの確認・修正を受けてから次へ進む
- 削除・大幅な構成変更は必ず理由を添えてユーザーに確認する

---

## ステップ 1: 変更スコープの把握

以下を順番に確認し、「何が変わったか」を把握する:

1. 最近完了した `.steering/` ドキュメントを確認する（実装済みフィーチャーの洗い出し）
2. `git log` で直近のコミット履歴を確認する

```bash
# 直近 20 件のコミットを確認
git log --oneline -20

# ステアリングドキュメントの一覧を確認
ls -lt .steering/
```

---

## ステップ 2: docs/ の現状確認

既存の `docs/` ファイルをすべて読み、コードの実態と照合する:

1. `docs/product-requirements.md` — フェーズ進捗・バックログに変化はあるか
2. `docs/architecture.md` — 新規モジュール・ディレクトリ・技術スタックの追加はあるか
3. `docs/development-guidelines.md` — 規約・ツール・手順に変化はあるか
4. `CLAUDE.md` — 「実装状況」セクションの「実装済み / 未実装」に変化はあるか

---

## ステップ 3: 更新計画をユーザーに提示・確認

ステップ 1〜2 の結果をもとに、以下の形式でユーザーに報告する:

```
## 更新が必要なドキュメント

### 更新あり
- [ ] docs/architecture.md — {理由: 例「DataCollection モジュールが新規追加された」}
- [ ] CLAUDE.md — {理由: 例「フェーズ 1 が実装済みに移行した」}

### 変更なし（スキップ）
- docs/product-requirements.md — フェーズ構成・バックログに変化なし
- docs/development-guidelines.md — 規約に変化なし

更新を進めてよいですか？
```

**ユーザーの承認を得てから次のステップへ進む**

---

## ステップ 4: ドキュメント更新（1 ファイルずつ確認）

承認されたファイルを順番に更新する。各ファイルの更新後にユーザーへ確認を取る。

### 更新の指針

| ドキュメント | 更新するべき内容 |
|---|---|
| `product-requirements.md` | フェーズのステータス変更、バックログへの移動・追加 |
| `architecture.md` | 新規ディレクトリ・モジュール・技術スタックの追加、削除されたモジュールの除去 |
| `development-guidelines.md` | ツール・コマンド・手順の変更、新ルールの追加 |
| `CLAUDE.md` | 「実装済み」「未実装」セクションの移動・追記 |

### 更新時の注意

- 既存の意思決定や制約（「〇〇は必須」「□□はやらない」等）は削除しない
- 事実の記録（何を作ったか）は追記のみ。過去の設計判断の書き換えは要確認
- `.steering/{date}-{feature}/design.md` に詳細設計があれば参照して反映する

---

## ステップ 5: 完了報告

全ファイルの更新完了後、以下を報告する:

```
## update-docs 完了

### 更新したファイル
- docs/architecture.md — {変更内容の要約}
- CLAUDE.md — {変更内容の要約}

### 次のアクション（あれば）
- {例: フェーズ 2 の設計を始める場合は /add-feature backtest-engine を実行}
```
